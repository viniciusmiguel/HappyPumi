# HappyPumi

An open-source reimplementation of the **Pulumi Cloud** API. The goal is wire-compatibility with the real
Pulumi service: a stock `pulumi` CLI should be able to `login`, set config, and run the full update lifecycle
(`up` / `preview` / `refresh` / `destroy`) against HappyPumi, plus the IDP features (orgs, RBAC, templates,
policy, deployments).

## Clean-room implementation (read this first)

HappyPumi is a **clean-room implementation** derived **only** from two public sources: the public **Pulumi
Cloud OpenAPI spec** (`pulumi-spec.json`) and the **Apache-2.0-licensed Pulumi CLI**
(`github.com/pulumi/pulumi`). Pulumi Cloud's server-side source is proprietary and is **not** used, referenced,
copied, or decompiled. All server behavior is inferred from the public contract and the observable behavior of
the open-source client, then implemented independently. Comply with Apache-2.0 (attribution/NOTICE) for
anything derived from the CLI, and do not introduce knowledge from Pulumi's proprietary internals. See
**ADR-0008** (`docs/adr/0008-clean-room-implementation.md`).

## What the contract is

The authoritative API surface is **`pulumi-spec.json`** (the Pulumi Cloud OpenAPI spec: ~588 operations, 14
areas). We are free to implement the server however we like, as long as we honor that HTTP contract — the
real Pulumi service/CLI are Go, but we share no code with them; we port behavior by reading the public spec
and the Apache-2.0 CLI's HTTP client (see clean-room note above).

- **`ENDPOINTS.md`** — the reverse-engineering roadmap: which endpoints to implement, in priority order, to
  get basic IaC + IDP working. Only ~161 of the 588 operations are ever called by the CLI; the rest are
  console/other-product surfaces. **Read this before deciding what to implement next.**
- **`docs/adr/`** — Architecture Decision Records (MADR format, one decision per file). See the index in
  `docs/adr/README.md`. Add a new ADR rather than rewriting an old one.

## Solution layout

| Project | Purpose |
|---|---|
| `HappyPumi.Api` | The API. One FastEndpoints endpoint class per spec operation, organized into area folders (`Stacks/`, `Organizations/`, `Deployments/`, …). DTOs live in `Contracts/`. |
| `HappyPumi.ServiceDefaults` | Shared Aspire defaults: OpenTelemetry, health checks, service discovery, HTTP resilience. The API calls `AddServiceDefaults()` / `MapDefaultEndpoints()`. |
| `HappyPumi.Api.Tests` | xUnit component tests (in-process via `WebApplicationFactory<ApiMarker>`) + the Aspire topology test. |
| `HappyPumi.Cli.IntegrationTests` | Drives the real `pulumi` CLI against a live HappyPumi over HTTPS (wire-compat). |
| `HappyPumi.AppHost` | .NET Aspire composition root: Postgres (+pgWeb), Dex (OIDC), and the API over HTTPS. `make dev`. |
| `Generator` (`PulumiApiGenerator`) | Code generator: reads the OpenAPI spec and emits the endpoint + contract scaffolding. |

## Key decisions (see `docs/adr/`)

- **0001** — .NET 10 / C# everywhere.
- **0002** — FastEndpoints (REPR pattern); one `Endpoint<TReq,TResp>` per spec `operationId`.
- **0003** — xUnit + `Aspire.Hosting.Testing`.
- **0004** — Ship as a Docker container (multi-stage, non-root, ports 8080/8081); Aspire for local dev.
- **0005** — PostgreSQL (relational + `jsonb` for checkpoints/exports/config). ORM choice deferred.
- **0006** — OpenTelemetry (OTLP → Aspire dashboard locally, configurable backend in prod).
- **0007** — OpenID Connect auth + per-org RBAC. Concrete IdP deferred.
- **0008** — Clean-room implementation from public sources only (OpenAPI spec + Apache-2.0 CLI).
- **0009** — Support both GitHub and Azure DevOps as first-class VCS providers (behind one `IVcsProvider` seam).
- **0010** — Audit every infrastructure-changing action to pluggable blob storage (Azure/AWS/generic, fail-closed).

## The generator workflow (important)

Endpoints and contracts are **generated** from the spec. Each generated endpoint is a stub whose `HandleAsync`
throws `NotImplementedException`; "implementing" an endpoint means replacing that body. The `operationId` is
the class/file name (e.g. operation `CreateUpdateForUpdate` → `Stacks/CreateUpdateForUpdateEndpoint.cs`).

Run the generator from the `Generator/` directory:

```bash
# defaults: fetches the live spec from api.pulumi.com, writes into ../../../../HappyPumi.Api
dotnet run
# or pin to the local spec:
dotnet run -- --spec ../pulumi-spec.json --out ../HappyPumi.Api --ns HappyPumi.Api
```

> ⚠️ **The generator overwrites every endpoint and contract file unconditionally** (`File.WriteAllText`, no
> existence check). Re-running it will clobber any hand-written `HandleAsync` implementations. Before
> regenerating, either commit your work, or change the generator to skip files that already exist / preserve
> handler bodies. Treat generated files marked `// <auto-generated />` with care.

## Build, run, test

A `Makefile` wraps the common flows (`make help` lists them):

```bash
make build             # dotnet build HappyPumi.slnx
make dev               # run the full Aspire topology + dashboard locally
make test              # unit + CLI integration
make test-unit         # fast in-process component tests (no CLI, no infra)
make test-integration  # drive the REAL pulumi CLI against a live HappyPumi (HTTPS)
make pulumi            # build the pulumi CLI + Go language host from ../pulumi
make certs             # create + trust the self-signed HTTPS dev certificate
make docker            # build the production container image
```

Or directly:

```bash
dotnet build HappyPumi.slnx
dotnet run --project HappyPumi.Api          # Swagger UI is enabled
dotnet run --project HappyPumi.AppHost      # full Aspire topology (Postgres + Dex + API) + dashboard
dotnet test HappyPumi.Api.Tests
```

## HTTPS / self-signed certificate (ADR-0007)

The dev/test environment runs over **HTTPS with the self-signed ASP.NET Core dev certificate**. Trust it
once with `make certs` (`dotnet dev-certs https --trust`). On Linux, clients trust it via OpenSSL's
`SSL_CERT_DIR` pointing at `~/.aspnet/dev-certs/trust`: the test hosts set this automatically at load
(`tests-support/DevCertTrust.cs`, linked into both test projects) and the Go `pulumi` CLI honours the same
env. Use host `localhost` (the cert's SAN), not `127.0.0.1`.

## Testing tooling (three layers)

1. **`HappyPumi.Api.Tests` — component tests.** Boot the real API in-process with
   `WebApplicationFactory<ApiMarker>` (see `HappyPumiApp`); fast, no sockets, no Docker. The bulk of
   endpoint/contract behavior is tested here. (`ApiMarker` is a public type in the API — we don't use
   `Program` because the AppHost also emits one and that would be ambiguous.)
2. **`HappyPumi.Cli.IntegrationTests` — wire-compatibility against the real CLI.** Launches the built API
   as an out-of-process **HTTPS** server on a free port (`HappyPumiServer`) and drives the actual `pulumi`
   binary (`PulumiCli`, isolated `PULUMI_HOME` + token, `SSL_CERT_DIR` for cert trust) against it.
   `make pulumi` / `tools/build-pulumi-cli.sh` builds the Apache-2.0 CLI **and** the Go language host from
   `../pulumi` into `.tools/bin/` (gitignored). Clean-room: the CLI is a black-box client (ADR-0008).
3. **Aspire topology test (`AspireTopologyTests`).** Boots the whole AppHost (Postgres + Dex + API) like
   `make dev` and asserts the API comes up healthy and serves `/api/user` over HTTPS. Docker-backed and
   slow (~20s); auto-skips when Docker is unavailable. Health probe is over the http endpoint (no cert
   trust needed); the assertion uses https (self-signed).

**No real infra needed.** The CLI's DIY/mock backends would *bypass* HappyPumi, so they can't validate
it. Instead: login / stack / config / export need no Pulumi program at all; the update lifecycle
(`up`/`preview`/`refresh`/`destroy`) is exercised by a **resourceless** Go program
(`HappyPumi.Cli.IntegrationTests/fixtures/empty-stack`) — it drives the full
`CreateUpdate → checkpoints → complete` cycle while provisioning nothing. The Go language host is built
locally, so this runs offline. Lifecycle tests are `[Skip]`ped until Tier-1 endpoints land (ENDPOINTS.md).

## Conventions for implementing an endpoint

1. Find its priority/tier in `ENDPOINTS.md`; implement Tier 0 → 1 first (login + the update lifecycle).
2. Replace the generated `AllowAnonymous()` stub with a real auth policy (per ADR-0007) — an endpoint isn't
   "done" while it's still anonymous.
3. Implement the lifecycle for the `update` kind first, then clone to `preview` / `refresh` / `destroy`
   (identical except the path segment; `preview` has no checkpoint endpoints).
4. Land at least a smoke test (route wired, contract serializes), then behavioral tests.
5. Never hand-edit `// <auto-generated />` files in a way the next generator run would silently undo without
   first solving the overwrite problem above.
